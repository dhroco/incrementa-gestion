Diferencia clave local ↔ nube:
  • BD: local = Auth Proxy (127.0.0.1); nube = conector de socket unix (/cloudsql/…). Ninguno usa la IP pública.
  • Secretos: local = set-env-local.sh; nube = Secret Manager (inyectados por Cloud Run).
  • GCS / firma: local = key JSON en disco; nube = identidad de la Runtime SA (ADC) firmando por IAM signBlob — por eso la SA tiene tokenCreator sobre sí misma.
  • Frontend: local = Vite dev server; nube = build estático servido por nginx.

4. CI/CD — flujo de despliegue Etapa B.5 + C + D pendientes

GitHub Actions se autentica a GCP sin llaves (Workload Identity Federation). Un push a la rama preprod dispara build de imágenes y deploy a Cloud Run.

sequenceDiagram autonumber actor Dev as Desarrollador participant GH as GitHub (rama preprod) participant WF as GitHub Actions (workflow) participant WIF as GCP Workload Identity Fed. participant DSA as Deployer SA participant AR as Artifact Registry participant CR as Cloud Run (be + fe) Dev->>GH: git push preprod GH->>WF: dispara workflow WF->>WIF: presenta token OIDC de GitHub Note over WIF: Verifica atributo
repository == owner/repo WIF-->>WF: credencial de corta vida (impersona Deployer SA) WF->>AR: docker build + push (backend y frontend)
frontend con build-args VITE_* WF->>CR: deploy backend
(--add-cloudsql-instances, secretos de SM,
runtime SA, env vars) WF->>CR: deploy frontend (nginx) CR->>AR: pull de imágenes CR-->>Dev: URLs de servicio (be + fe)

Por qué WIF (keyless) y no una llave de SA

  • No hay que subir un JSON de service account a GitHub Secrets (nada que se filtre o rote).
  • La confianza se limita por atributo: solo owner/repo (y opcionalmente la rama) puede autenticarse.
  • Las credenciales que recibe el workflow son efímeras (minutos), no una llave permanente.
Pendiente para armar B.5: falta el owner/repo de GitHub para crear el provider WIF y el binding. La Deployer SA tendrá artifactregistry.writer, run.admin y iam.serviceAccountUser (sobre la Runtime SA).

5. Flujo de autenticación — MSAL + identidad interna por email

Aplica igual en local y en la nube (cambian solo las URLs). El backend confía en el token de Entra ID, pero la identidad interna se resuelve por email, no por el sub del IdP.

sequenceDiagram autonumber actor U as Usuario participant FE as Frontend (MSAL) participant AZ as Microsoft Entra ID participant BE as Backend Express participant DB as Cloud SQL (user_profile) U->>FE: abre la app FE->>AZ: Authorization Code + PKCE AZ-->>FE: id_token + access_token (JWT) FE->>BE: GET sesión (Authorization: Bearer JWT) BE->>AZ: valida firma/issuer/audience (OIDC discovery) AZ-->>BE: token válido (claims: email, oid…) BE->>DB: SELECT user_profile WHERE lower(email)=lower(claim.email) DB-->>BE: perfil + is_active + avatar_gcs_path BE->>BE: arma permisos (CASL) + firma URL de avatar BE-->>FE: sesión enriquecida (perfil, permisos, avatarUrl) FE-->>U: app cargada (menús según rol)
Por qué identidad por email: desacopla la app del sub del proveedor. Si mañana cambia el IdP (o el tenant del cliente en prod), el usuario sigue mapeando por su email mediante un índice único parcial sobre LOWER(email). El alta de usuarios valida además que el email exista en el tenant vía Microsoft Graph app-only.

9. Ideas clave para recordar

  • Una sola BD. Local y nube apuntan a la misma incrementa-db (us-central1). Cambia el método de conexión, no la base.
  • Sin llaves en la nube. Runtime = ADC de la SA; deploy = WIF keyless. Las únicas llaves en disco están en local (proxy + GCS key), fuera de git.
  • Secretos por entorno. Local: set-env-local.sh. Nube: Secret Manager. Nunca en el repo.
  • Firmar URLs necesita una identidad que pueda firmar. Local con key; nube con tokenCreator (IAM signBlob). El ADC de usuario no firma.
  • Identidad por email desacopla del IdP → migrar de tenant/proveedor no rompe los usuarios.
  • Dos identidades de máquina con roles mínimos: runtime (correr) vs deployer (desplegar).
