MCP Remoto — Opciones de Autenticación

Para exponer el MCP de incrementa-gestion a Claude Desktop y Claude.ai (web), hace falta un servidor MCP remoto con OAuth. El problema: los clientes de Claude requieren Dynamic Client Registration (DCR), y Microsoft Entra ID no lo soporta. Por eso necesitamos una capa intermedia. Estas son las 3 formas de resolverlo.

Lo común a las 3 opciones

Lo único que cambia entre opciones es la caja de autenticación. El resto es idéntico: un servicio nuevo en Cloud Run que reusa la lógica del backend, con permisos por usuario.

En las 3, el MCP recibe la identidad del usuario (email) ya autenticada, resuelve su perfil (user_profile) y aplica sus permisos CASL —igual que la API—. Se elimina el "usuario técnico god-mode" actual.

Con un plan Claude Team/Enterprise, el administrador de la organización provisiona el conector MCP centralmente a través de Entra ID (Enterprise-Managed Authorization). Los usuarios quedan conectados al iniciar sesión, sin OAuth por-app. Es la experiencia más fluida y la más alineada con un IdP corporativo.

Recomendación. Para arrancar con menor riesgo y sin construir infraestructura de seguridad crítica, la Opción 1 es el mejor punto de partida —funciona en Desktop y web y federa a Entra—. Si la organización ya está pensando en Claude Enterprise, la Opción 3 da la mejor experiencia a futuro (incluso podrían empezar con la 1 y migrar a la 3). La Opción 2 solo si hay una exigencia clara de no depender de un SaaS y capacidad para mantener OAuth en el tiempo. En las tres, el trabajo del MCP en sí (transporte remoto + permisos por usuario) es el mismo.

Documento de decisión — incrementa-gestion · MCP remoto. La elección de esta capa define solo el "borde de autenticación"; el resto del diseño (servicio Cloud Run incrementa-mcp, transporte Streamable HTTP, refactor de permisos por usuario) es común y se implementa vía OpenSpec.
