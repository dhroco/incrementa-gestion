Manual de configuración · Google Cloud (GCP) — Producción

Sistema incrementa-gestion · Para el administrador del proyecto GCP del cliente · Cloud Run + Cloud SQL + GCS + Artifact Registry + Secret Manager + CI/CD keyless (WIF)

0. Propósito y a quién va dirigido

Deja operativo el backend y el frontend de incrementa-gestion en el proyecto GCP de producción del cliente, con despliegue automatizado desde GitHub Actions sin llaves (Workload Identity Federation).

Lo ejecuta alguien con permisos de administrador/propietario sobre el proyecto GCP del cliente (o el equipo de desarrollo junto al admin). Requiere valores generados en el Manual de Azure Entra ID (Tenant ID, Client ID, Client secret).

Arquitectura destino (idéntica a pre-prod): dos servicios Cloud Run (backend Express + frontend nginx), una instancia Cloud SQL (PostgreSQL 16), un bucket GCS, un repo Docker en Artifact Registry, secretos en Secret Manager, dos service accounts (runtime y deployer) y WIF para el CI. Todo sin claves JSON descargadas.

1. Prerrequisitos

  • gcloud CLI instalado y autenticado: gcloud auth login.
  • Un proyecto GCP de producción del cliente, con facturación habilitada, y permisos de Owner (o roles equivalentes: Service Usage Admin, Cloud SQL Admin, Storage Admin, Artifact Registry Admin, Secret Manager Admin, IAM Admin, Cloud Run Admin).
  • El repositorio GitHub del proyecto (<OWNER/REPO>) y la rama de producción (ej. prod).
  • Los valores del Manual de Azure: <TENANT_ID>, <CLIENT_ID>, <CLIENT_SECRET>.
  • La API key de Resend de prod (<RESEND_API_KEY>) y el remitente (<RESEND_FROM>).

2. Variables — defínalas una vez en su terminal

Todos los comandos siguientes usan estas variables. Reemplace los valores <…> por los reales del cliente y pegue el bloque en su shell (bash/zsh).

export PROJECT_ID="<PROYECTO_PROD_DEL_CLIENTE>"
export REGION="us-central1"                 # o la región que prefiera el cliente
export REPO="<OWNER/REPO>"                   # ej. cliente-org/incrementa-gestion
export PROD_BRANCH="prod"                    # rama que dispara el deploy de prod
export DB_INSTANCE="incrementa-db"
export DB_NAME="incrementa"
export DB_PASSWORD="<PASSWORD_FUERTE_BD>"
export BUCKET="<BUCKET_PROD>"                # ej. incrementa-contratos-prod (único global)
export AR_REPO="incrementa"

# Valores desde el Manual de Azure:
export AZURE_TENANT_ID="<TENANT_ID>"
export AZURE_CLIENT_ID="<CLIENT_ID>"
export AZURE_CLIENT_SECRET="<CLIENT_SECRET>"

# Email (Resend):
export RESEND_API_KEY="<RESEND_API_KEY>"
export RESEND_FROM="<RESEND_FROM>"          # ej. contratos@dominio-cliente.com

gcloud config set project "$PROJECT_ID"
Guarde también el project number (se usa en WIF): 
export PROJECT_NUMBER="$(gcloud projects describe "$PROJECT_ID" --format='value(projectNumber)')"
echo "$PROJECT_NUMBER"

4. Cloud SQL (PostgreSQL 16)

  1. Crear la instancia (ajuste tier según la carga de prod; db-f1-micro es mínimo, para prod considere db-custom-1-3840 o superior): 
    gcloud sql instances create "$DB_INSTANCE" \
  --database-version=POSTGRES_16 \
  --region="$REGION" \
  --tier=db-custom-1-3840 \
  --storage-auto-increase
  2. Password del usuario postgres y base de datos: 
    gcloud sql users set-password postgres \
  --instance="$DB_INSTANCE" --password="$DB_PASSWORD"

gcloud sql databases create "$DB_NAME" --instance="$DB_INSTANCE"
  3. Anote el connection name (se usa en el socket y en Cloud Run): 
    export CONNECTION_NAME="$(gcloud sql instances describe "$DB_INSTANCE" --format='value(connectionName)')"
echo "$CONNECTION_NAME"   # -> PROJECT_ID:REGION:incrementa-db
Cargar el esquema (migraciones): la base arranca vacía. El equipo de desarrollo debe correr las migraciones Knex contra esta instancia una vez (vía Cloud SQL Auth Proxy con DATABASE_URL apuntando al proxy, ejecutando el script de migración del backend). Esto se coordina con el equipo de desarrollo; no forma parte de los comandos de este manual.
Respaldos: para producción habilite backups automáticos y point-in-time recovery (--backup-start-time, --enable-point-in-time-recovery) según la política del cliente.

5. Cloud Storage (bucket de documentos y avatares)

gcloud storage buckets create "gs://$BUCKET" \
  --location="$REGION" \
  --uniform-bucket-level-access

El nombre del bucket es único a nivel global; si está tomado, elija otro y actualice $BUCKET.

6. Artifact Registry (repo Docker)

gcloud artifacts repositories create "$AR_REPO" \
  --repository-format=docker \
  --location="$REGION" \
  --description="Imagenes Cloud Run incrementa-gestion (prod)"

Ruta del repo (se usará en el CI): $REGION-docker.pkg.dev/$PROJECT_ID/$AR_REPO

7. Secret Manager (3 secretos)

Se referencian por nombre desde Cloud Run; sus valores nunca van al repositorio. Se usa printf (no echo) para no agregar un salto de línea que corrompa el secreto.

7.1 DATABASE_URL (con socket del conector — sin IP, sin SSL)

printf '%s' "postgresql://postgres:$DB_PASSWORD@/$DB_NAME?host=/cloudsql/$CONNECTION_NAME" \
  | gcloud secrets create DATABASE_URL --data-file=- --replication-policy=automatic

7.2 GRAPH_CLIENT_SECRET (desde el Manual de Azure)

printf '%s' "$AZURE_CLIENT_SECRET" \
  | gcloud secrets create GRAPH_CLIENT_SECRET --data-file=- --replication-policy=automatic

7.3 RESEND_API_KEY

printf '%s' "$RESEND_API_KEY" \
  | gcloud secrets create RESEND_API_KEY --data-file=- --replication-policy=automatic
Para rotar un secreto en el futuro: printf '%s' "<nuevo>" | gcloud secrets versions add <NOMBRE> --data-file=- (Cloud Run usa :latest).

8. Runtime Service Account (identidad de los contenedores)

export RUNTIME_SA="incrementa-run-sa@$PROJECT_ID.iam.gserviceaccount.com"

gcloud iam service-accounts create incrementa-run-sa \
  --display-name="Incrementa Cloud Run runtime"

# Conectar a Cloud SQL
gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:$RUNTIME_SA" --role="roles/cloudsql.client"

# Leer secretos
gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:$RUNTIME_SA" --role="roles/secretmanager.secretAccessor"

# Leer/escribir en el bucket
gcloud storage buckets add-iam-policy-binding "gs://$BUCKET" \
  --member="serviceAccount:$RUNTIME_SA" --role="roles/storage.objectAdmin"

# Firmar URLs de GCS (avatares/documentos) vía IAM signBlob — sobre sí misma
gcloud iam service-accounts add-iam-policy-binding "$RUNTIME_SA" \
  --member="serviceAccount:$RUNTIME_SA" --role="roles/iam.serviceAccountTokenCreator"
No omita el último binding (serviceAccountTokenCreator): en Cloud Run las URLs firmadas de GCS se generan con IAM signBlob usando la identidad de la runtime SA. Sin este rol, la carga de avatares/documentos firmados falla con 403.

9. Deployer SA + Workload Identity Federation (CI keyless)

GitHub Actions se autentica sin llaves: un pool OIDC confía en los tokens de Actions, restringido a su repo y su rama de prod, y permite impersonar la deployer SA.

9.1 Deployer SA + roles

export DEPLOYER_SA="incrementa-deploy-sa@$PROJECT_ID.iam.gserviceaccount.com"

gcloud iam service-accounts create incrementa-deploy-sa \
  --display-name="Incrementa CI/CD deployer"

gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:$DEPLOYER_SA" --role="roles/artifactregistry.writer"

gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:$DEPLOYER_SA" --role="roles/run.admin"

# Permitir a la deployer "actuar como" la runtime SA (para desplegar Cloud Run con esa identidad)
gcloud iam service-accounts add-iam-policy-binding "$RUNTIME_SA" \
  --member="serviceAccount:$DEPLOYER_SA" --role="roles/iam.serviceAccountUser"

9.2 WIF pool + provider (restringido a repo AND rama)

gcloud iam workload-identity-pools create github-pool \
  --location="global" --display-name="GitHub Actions pool"

gcloud iam workload-identity-pools providers create-oidc github-provider \
  --location="global" \
  --workload-identity-pool="github-pool" \
  --display-name="GitHub provider" \
  --issuer-uri="https://token.actions.githubusercontent.com" \
  --attribute-mapping="google.subject=assertion.sub,attribute.repository=assertion.repository,attribute.ref=assertion.ref" \
  --attribute-condition="assertion.repository=='$REPO' && assertion.ref=='refs/heads/$PROD_BRANCH'"

9.3 Binding: permitir al repo impersonar la deployer SA

gcloud iam service-accounts add-iam-policy-binding "$DEPLOYER_SA" \
  --role="roles/iam.workloadIdentityUser" \
  --member="principalSet://iam.googleapis.com/projects/$PROJECT_NUMBER/locations/global/workloadIdentityPools/github-pool/attribute.repository/$REPO"
La condición del provider ya exige repo AND rama ($PROD_BRANCH), así que aunque el binding sea por repo, solo tokens de esa rama pasan. Defensa en profundidad.

10. Datos a recopilar para el CI (entregar al equipo de desarrollo)

echo "PROJECT_ID       = $PROJECT_ID"
echo "PROJECT_NUMBER   = $PROJECT_NUMBER"
echo "REGION           = $REGION"
echo "AR_HOST          = $REGION-docker.pkg.dev/$PROJECT_ID/$AR_REPO"
echo "CONNECTION_NAME  = $CONNECTION_NAME"
echo "BUCKET           = $BUCKET"
echo "RUNTIME_SA       = $RUNTIME_SA"
echo "DEPLOYER_SA      = $DEPLOYER_SA"
echo "WIF_PROVIDER     = projects/$PROJECT_NUMBER/locations/global/workloadIdentityPools/github-pool/providers/github-provider"
Lado desarrollo: con estos datos, el equipo crea el workflow de prod (ej. .github/workflows/deploy-prod.yml, análogo al de pre-prod pero disparado en la rama $PROD_BRANCH y apuntando a estos recursos), con ENVIRONMENT=prod y las variables OIDC/Graph del cliente. Los secretos (DATABASE_URL, GRAPH_CLIENT_SECRET, RESEND_API_KEY) se referencian por nombre desde Secret Manager.

11. Primer deploy + wire-up con Azure

  1. El equipo de desarrollo hace push a la rama $PROD_BRANCH → el pipeline construye y despliega backend y frontend, y setea el CORS_ORIGIN del backend con la URL del frontend.
  2. Tomar la URL del frontend que imprime el pipeline y entregarla al administrador de Azure → registra el redirect URI (SPA) (paso 7 del Manual de Azure).
  3. Variables de entorno de backend en Cloud Run (prod) que el workflow debe setear (referencia):
    VariableValor
    ENVIRONMENTprod
    OIDC_ISSUER_URLhttps://login.microsoftonline.com/<TENANT_ID>/v2.0
    OIDC_AUDIENCE, GRAPH_CLIENT_ID<CLIENT_ID>
    GRAPH_TENANT_ID<TENANT_ID>
    GCS_BUCKET<BUCKET>
    RESEND_FROM_EMAIL<RESEND_FROM>
    Secretos (--set-secrets)DATABASE_URL, GRAPH_CLIENT_SECRET, RESEND_API_KEY (por nombre)
    NO setearPORT, GOOGLE_APPLICATION_CREDENTIALS, PGSSLMODE
  4. El backend se despliega con --add-cloudsql-instances "$CONNECTION_NAME" y --service-account "$RUNTIME_SA". El frontend también con --service-account "$RUNTIME_SA".

Verificación (smoke)

  • GET <BACKEND_URL>/health responde {"status":"ok","environment":"prod"}
  • Login con un usuario del tenant carga la sesión (perfil + permisos) sin errores
  • Los logs de Cloud Run no muestran ECONNREFUSED 127.0.0.1:5432 (confirma conexión por socket)
  • Alta de usuario de plataforma valida contra Graph

12. Seguridad + checklist final

  • Sin llaves: runtime = identidad de la SA (ADC); deploy = WIF keyless. No se descargan claves JSON.
  • Menor privilegio: runtime SA y deployer SA con solo los roles listados. WIF restringido a <OWNER/REPO> + rama $PROD_BRANCH.
  • Secretos: solo en Secret Manager; el DATABASE_URL usa el socket del conector (no IP pública). Rote GRAPH_CLIENT_SECRET cuando Azure lo renueve.
  • Cloud SQL: sin IP pública necesaria (se conecta por el conector). Habilite backups/PITR para prod.
  • Cloud Run: ambos servicios son públicos (--allow-unauthenticated); el backend protege sus rutas con validación de JWT.
