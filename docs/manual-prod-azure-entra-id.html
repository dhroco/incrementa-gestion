Manual de configuración · Microsoft Entra ID — Producción

Sistema incrementa-gestion · Para el administrador del tenant Entra ID del cliente · Autenticación OIDC (login) + Microsoft Graph (validación de usuarios)

0. Propósito y a quién va dirigido

Este manual describe, paso a paso, cómo dejar operativa la identidad del sistema incrementa-gestion en el tenant de Microsoft Entra ID del cliente, para el ambiente de producción.

Lo ejecuta un administrador con permisos suficientes en el tenant (rol Administrador de aplicaciones o Administrador global; se requiere poder conceder consentimiento de administrador). Al terminar, entregará al equipo de desarrollo un conjunto de valores (sección 8) que se cargarán en GCP.

Relación con el manual de GCP: el CLIENT_SECRET y los IDs que se generan aquí se cargan luego en GCP (Secret Manager y variables de entorno de Cloud Run). El redirect URI del frontend (paso 4) solo se conoce después del primer despliegue en GCP, así que ese paso se completa al final, coordinado con el manual de GCP.

1. Prerrequisitos

  • Acceso al portal.azure.com con un rol que permita crear App Registrations y conceder consentimiento de administrador (Administrador de aplicaciones o Administrador global).
  • Conocer el dominio principal del tenant del cliente (ej. cliente.onmicrosoft.com o su dominio corporativo).
  • Un usuario de prueba real dentro del tenant para validar el login al final.

3. Crear la App Registration

  1. Portal Azure → Microsoft Entra ID → App registrations → + New registration.
  2. Name: incrementa-gestion-prod (o el nombre que defina el cliente).
  3. Supported account types: seleccione Accounts in this organizational directory only (single-tenant). El sistema es para usuarios del tenant del cliente.
  4. Redirect URI: déjelo vacío por ahora (se configura en el paso 7).
  5. Register.
  6. En la pantalla Overview de la app, copie y anote en la tabla del paso 2:
    • Application (client) ID → <CLIENT_ID>
    • Directory (tenant) ID → <TENANT_ID>

4. Exponer una API — scope access_as_user

El frontend solicita un token para esta API; el backend valida ese token. Necesitamos definir el Application ID URI y un scope.

  1. En la app → menú Expose an API.
  2. Application ID URI: pulse Add / Set y acepte el valor por defecto api://<CLIENT_ID>. Anótelo como <API_APP_ID_URI>.
  3. + Add a scope:
    • Scope name: access_as_user
    • Who can consent: Admins and users
    • Admin consent display name: Acceder como usuario
    • Admin consent description: Permite a la aplicación acceder a la API en nombre del usuario.
    • State: Enabled
  4. Add scope. El scope completo queda como api://<CLIENT_ID>/access_as_user.

5. Permiso de Microsoft Graph + consentimiento de administrador

El backend valida, al dar de alta usuarios, que el email exista en el tenant. Usa Graph en modo app-only (client credentials), por lo que requiere un permiso de tipo Aplicación.

  1. En la app → menú API permissions → + Add a permission.
  2. Elija Microsoft Graph → Application permissions (NO "Delegated").
  3. Busque y marque User.Read.All. → Add permissions.
  4. De vuelta en la lista, pulse Grant admin consent for <organización> y confirme.
    El estado de User.Read.All debe quedar en verde: Granted.
Importante: debe ser de tipo Aplicación, no Delegado. Un token de client-credentials solo transporta permisos de aplicación; con Delegado, Graph responde 403 Insufficient privileges. Y el consentimiento de admin es obligatorio (un usuario normal no puede otorgar User.Read.All).

6. Crear el secreto de cliente (client secret)

  1. En la app → menú Certificates & secrets → pestaña Client secrets → + New client secret.
  2. Description: incrementa-backend-graph.
  3. Expires: según política del cliente (recomendado 6–12 meses; anote la fecha para renovarlo a tiempo).
  4. Add.
  5. Copie inmediatamente el valor (columna «Value») y guárdelo como <CLIENT_SECRET>.
    Solo se muestra una vez; si lo pierde, deberá generar otro.
Manejo del secreto: es una credencial sensible. No lo envíe por correo/chat sin cifrar. Idealmente, el equipo de desarrollo lo carga directamente en GCP Secret Manager (o el propio admin lo hace). Nunca queda en el repositorio.

7. Redirect URI de la SPA (se completa tras el deploy en GCP)

El login del frontend (MSAL, Authorization Code + PKCE) redirige de vuelta a la URL del frontend. Esa URL se conoce después del primer despliegue en Cloud Run (ver manual de GCP).

  1. Cuando el equipo de GCP le entregue la <FRONTEND_URL> (ej. https://incrementa-frontend-xxxx-uc.a.run.app o el dominio propio de prod):
  2. App → Authentication → + Add a platform → Single-page application (SPA).
  3. En Redirect URIs agregue exactamente la <FRONTEND_URL> (sin barra final ni ruta). Ej.: https://app.cliente.com
  4. Configure / Save.
Nota: debe ser plataforma SPA (no "Web"). No se requiere activar flujos implícitos: el sistema usa Authorization Code + PKCE. Si en el futuro cambia el dominio del frontend, agregue el nuevo URI aquí.

9. Verificación (al final del wire-up)

Una vez que GCP esté desplegado y el redirect URI registrado (paso 7):

  • Un usuario de prueba del tenant puede iniciar sesión en <FRONTEND_URL> con su cuenta Microsoft.
  • Tras el login, la aplicación carga (perfil y menús) sin error de redirect URI mismatch (AADSTS50011).
  • El alta de un usuario de plataforma con un email inexistente en el tenant es rechazada (valida contra Graph); con un email existente, procede. Esto confirma que User.Read.All + consentimiento funcionan.

10. Notas de seguridad

  • Menor privilegio: el único permiso de aplicación es User.Read.All. No agregue permisos adicionales salvo necesidad justificada.
  • Rotación del secreto: anote la fecha de expiración del client secret y renuévelo antes de que caduque (genere uno nuevo, actualícelo en GCP Secret Manager, y elimine el viejo).
  • Single-tenant: la app acepta solo cuentas del directorio del cliente.
  • No versionar secretos: el client secret nunca va al repositorio; vive solo en Secret Manager (GCP).

Checklist final

  • App Registration creada (single-tenant) y anotados Client ID + Tenant ID
  • Application ID URI + scope access_as_user creados
  • Permiso User.Read.All (Aplicación) agregado y con consentimiento de admin
  • Client secret creado, copiado y entregado de forma segura
  • Redirect URI (SPA) registrado con la URL del frontend de prod
  • Valores entregados al equipo de desarrollo (sección 8)
  • Verificación de login + Graph superada
